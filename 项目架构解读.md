# Demo01 项目架构解读

> 面向有 Java / Python 经验、刚接触 Godot 的开发者。
> 本文基于项目当前代码(`project.godot`、各 `.gd`、`.tscn`)整理。

---

## 0. 一句话心法

> **`.gd` 脚本自己不会"运行",它是节点的"行为组件"。节点进了场景树,脚本才被激活。**

Java / Python 有固定的 `main()` 入口、由你驱动控制流;Godot **没有 main()**,入口是「场景树(Scene Tree)」,由**引擎驱动生命周期回调**。你只是"填回调",引擎负责调度。

---

## 1. Godot 运行模型 vs Java / Python

| | Java / Python | Godot |
|---|---|---|
| 入口 | 固定的 `main()` 函数 | **没有 main()**,入口是场景树 |
| 控制流 | **你调用代码**(线性往下走) | **引擎调用你**(回调驱动) |
| 对象生命周期 | 手动 `new`、手动管理 | 引擎按节点树自动管理 |
| 全局共享对象 | 单例 / 依赖注入 | **autoload**(自动注册的全局单例) |

### 启动流程(对应 `project.godot`)

1. 引擎启动 → 读取 `[autoload]` 段,把三个脚本实例化成节点,挂到 `/root` 下,**常驻整个游戏**。
2. 读取 `run/main_scene="res://scenes/main.tscn"`,实例化后挂成 `/root/Main`。
3. 此后引擎**每一帧自动调用**每个在树节点的 `_process(delta)`,节点进树时自动调用 `_ready()`……

> 类比:Godot 引擎 ≈ Spring 容器;`_ready()` ≈ `@PostConstruct`;autoload ≈ 容器启动时自动创建的单例 Bean。

## 2. autoload:全局单例

`project.godot` 中的注册:
```ini
[autoload]
game_manager="*res://autoload/game_manager.gd"
enemy_spawner="*res://autoload/enemy_spawner.gd"
upgrade_manager="*res://autoload/upgrade_manager.gd"
```
- `*` 表示"把脚本实例化成一个常驻节点"。
- **注册的名字会自动成为全局可访问的标识符**。任何脚本都能直接写 `game_manager.xxx`,**不需要 import、不需要传引用**。

这就是项目里到处能写 `game_manager.game_time` 的原因——它不是"没有互相引用",而是**用全局单例名代替了 Java 里的 import / 依赖注入**。

### autoload 之间的依赖关系

```
        game_manager   ←── 中央状态(时间、击杀、player 引用)
         ▲       ▲
         │       │   (通过全局单例名读写)
  enemy_spawner   upgrade_manager
```

证据:
- `enemy_spawner.gd`:`game_manager.game_over`、`game_manager.player`、`game_manager.game_time`
- `upgrade_manager.gd`:`game_manager.player.might`、`game_manager.player.max_hp`……

`game_manager` 是"中央数据",另外两个是"系统",都读写它。

### autoload ↔ main 场景的互访

autoload 用**绝对路径**反向找到 main 场景里的节点:
```gdscript
# enemy_spawner.gd:27-28
enemies_container = get_node("/root/Main/GameWorld/Enemies")
xp_gems_container = get_node("/root/Main/GameWorld/XPGems")
```
> 注意:autoload 的 `_ready()` 比 main 场景的节点**先执行**,所以代码里有 `await get_tree().process_frame`,等一帧让 main 场景先建好,再取节点引用。

## 3. 脚本与节点的绑定:发生在 `.tscn` 里

脚本如何挂到节点上?看 `player.tscn`:
```
[node name="Player" type="CharacterBody2D"]
script = ExtResource("script_player")        ← 挂 player_controller.gd

[node name="WeaponPistol" type="Node" parent="."]
script = ExtResource("script_weapon")        ← 挂 weapon_pistol.gd
```
每个 `.tscn` 用 `script = ...` 声明节点穿哪件"行为衣服"。**先有节点,脚本才有载体。**

### preload ≠ 运行
```gdscript
const PlayerScene = preload("res://scenes/player.tscn")   # 只加载"模板"
var player = PlayerScene.instantiate()                     # 这一步才"创建实例"
game_world.add_child(player)                              # 进树 → 触发 _ready
```
- `preload` ≈ Java 里把 `Class` 加载进内存。
- `.instantiate()` ≈ `new`。
- `add_child()` ≈ 把对象挂进容器,引擎才开始调度它。

## 4. 全部 12 个脚本的运行时机

按"生命周期"分三类,正好对应 Java 里"单例 / 启动对象 / 临时对象"。

### ① 启动即创建,常驻到游戏结束(类似容器单例)

| 脚本 | 绑定节点 | 谁把它放进树 | 运行时机 |
|---|---|---|---|
| `autoload/game_manager.gd` | `/root/GameManager` | 引擎(project.godot) | 启动(最早) |
| `autoload/enemy_spawner.gd` | `/root/EnemySpawner` | 引擎 | 启动 |
| `autoload/upgrade_manager.gd` | `/root/UpgradeManager` | 引擎 | 启动 |
| `scripts/main.gd` | `main.tscn` 根节点 `Main` | 引擎(主场景) | 启动 |
| `ui/hud_controller.gd` | `hud.tscn`(main.tscn:64 `instance`) | main.tscn 加载连带 | 启动 |
| `ui/level_up_controller.gd` | `level_up_panel.tscn`(main.tscn:66) | 同上 | 启动 |
| `ui/game_over_controller.gd` | `game_over_panel.tscn`(main.tscn:68) | 同上 | 启动 |

> 三个 UI 在 `main.tscn` 里用 `instance=ExtResource(...)` **直接写死**,加载 main 时"打包一起出生",无需代码 instantiate。

### ② 启动时创建一次(类似启动时 `new` 核心对象)

| 脚本 | 绑定节点 | 谁实例化它 | 运行时机 |
|---|---|---|---|
| `scripts/player/player_controller.gd` | `player.tscn` 根节点 `Player` | `main.gd:13-14` 代码 `instantiate`+`add_child` | 启动后第一帧 |
| `scripts/weapon/weapon_pistol.gd` | `player.tscn` 子节点 `WeaponPistol` | Player 实例化时连带(父子关系) | 与 Player 同时 |

> `WeaponPistol` 是 `Player` 的子节点,实例化 Player 时引擎自动连带建出子节点树——组合关系,父 new 出来,子跟着 new。

### ③ 运行中反复 创建 → 销毁(类似方法里频繁 `new` 的临时对象)

| 脚本 | 绑定节点 | 谁实例化它 | 运行时机 |
|---|---|---|---|
| `scripts/enemy/enemy_base.gd`（+ 5 个子类 `enemy_<swarmer/normal/tank/shooter/boss>.gd`） | `enemy_*.tscn` 根节点 | `enemy_spawner._spawn_at()`（经预警回调） | 游戏中持续刷怪 |
| `scripts/projectile/projectile_base.gd` | `projectile.tscn` 根节点 | `weapon_pistol._fire()` | 武器冷却到点发射 |
| `effects/xp_gem.gd` | `xp_gem.tscn` 根节点 | `enemy_spawner._on_enemy_died()` | 敌人死亡瞬间 |

> 这类对象不仅被创建,还会被 `queue_free()` 销毁(如 `enemy_spawner.gd:129` 清理远处敌人)。是"短命对象"。

## 5. 完整启动时间线

```
1. 引擎读 project.godot → 实例化 3 个 autoload 进 /root
   ├─ game_manager._ready()     ← player 还是 null
   ├─ enemy_spawner._ready()    ← preload 敌人模板,await 一帧
   └─ upgrade_manager._ready()

2. 引擎加载 main.tscn → 构建 Main 节点树(连带 UI 三个)
   ├─ main.gd._ready()          ← 这里 instantiate 出 Player
   ├─ hud / level_up / game_over 的 _ready()
   └─ Player 进树 →
        ├─ player_controller._ready()
        └─ WeaponPistol 进树 → weapon_pistol._ready()

3. 之后每帧,引擎自动调所有在树节点的 _process(delta):
   ├─ game_manager._process()     ← 计时、发 time_tick 信号
   ├─ enemy_spawner._process()    ← 刷怪(instantiate enemy)
   ├─ weapon_pistol._process()    ← 冷却到了就 _fire()(instantiate projectile)
   └─ player_controller._process()← 移动 ……
```

## 7. `_ready` vs `_process`:两种"运行"

| 回调 | 触发时机 | 次数 | Java 类比 |
|---|---|---|---|
| `_ready()` | 节点**进树时** | 一次 | 构造器 / `@PostConstruct` |
| `_process(delta)` | 节点**在树期间每帧** | 每帧 | 主循环里的 `update()` |
| `_physics_process(delta)` | 物理帧(固定步长) | 每物理帧 | 物理更新 |

> 第 4 节表格里的"运行时机"主要指 `_ready` 触发点;而刷怪、计时、发射等持续行为靠的是每帧的 `_process`。

## 8. 节点通信:信号(Signal)

Godot 节点之间通过**信号**通信,类比 Java 的**事件监听器 / 观察者模式**。

机制:
```gdscript
# 定义方(发送者):声明信号
signal enemy_died(pos: Vector2)

# 接收方:连接信号到回调
enemy.enemy_died.connect(_on_enemy_died)

# 发送方:触发
enemy_died.emit(global_position)
```

本项目已观察到的信号:
- `game_manager`:`time_tick(delta)`、`kill_counted()`、`game_ended(won)`(全局事件总线)
- `upgrade_manager`:`xp_changed(...)`、`level_up()`
- `player.damaged` → 连到 `hud._on_player_damaged`(`main.gd:17`)
- `enemy.enemy_died(pos)` → 连到 `enemy_spawner._on_enemy_died`(`enemy_spawner.gd:80`)

> 信号让节点之间**松耦合**:发送者不需要知道谁在听,只管 `emit`;接收者 `connect` 自己关心的信号。这比直接互相调用方法更解耦。

## 9. 物理碰撞层(layer_names)

`project.godot` 中定义,用于决定"谁能撞到谁":

| 层 | 名称 | 用途 |
|---|---|---|
| 1 | Player | 玩家 |
| 2 | Enemies | 敌人 |
| 3 | Projectiles | 子弹 |
| 4 | XPGems | 经验宝石 |
| 5 | PlayerPickup | 玩家拾取区域 |
| 6 | EnemyDamage | 敌人伤害区域 |
| 7 | Walls | 墙壁 |

节点的 `collision_layer` 表示"我在哪一层",`collision_mask` 表示"我能撞到哪几层"。
