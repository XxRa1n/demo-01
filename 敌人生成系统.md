# 敌人生成系统

> 本文档记录 demo-01 当前的敌人方案：**环形生成 + 多类型（每类一个独立脚本/场景）+ 波次时间曲线（含定时 Boss 波）+ 敌人间软分离（防重叠）**。
> 生成相关数据驱动自 `autoload/enemy_spawner.gd` 顶部的注册表（`KIND_SCENES`/`KIND_SCRIPTS`）与波次表（`WAVES`/`BOSS_WAVES`）；每种敌人自己的数值写在它自己的脚本里（`const CONFIG`，单一数据源），软分离参数见 §5。
>
> ⚠️ 注：`WAVES` / `BOSS_WAVES` / 生成率公式等**玩法调参**会随平衡频繁改动，下文示例数值可能与代码有出入，以 `enemy_spawner.gd` 顶部实际常量为准。本文档主要保证「架构」描述准确。

---

## 1. 概览

| 组件 | 职责 | 关键文件 |
|------|------|----------|
| **环形生成** | 决定敌人"在哪生成"——以玩家为圆心的屏外圆环上随机取点 | `autoload/enemy_spawner.gd` → `get_spawn_position_ring()` |
| **多类型** | 决定敌人"是什么样/多强"——5 种敌人，**每种一个独立脚本 + 独立场景**；数值写在各自脚本的 `const CONFIG` | `scripts/enemy/enemy_*.gd` + `scenes/enemy_*.tscn`（见 §2.2） |
| **波次曲线** | 决定"什么时候生成什么/多密"——按游戏时间分阶段，定时触发 Boss 波 | `WAVES` / `BOSS_WAVES` 表 |
| **软分离** | 决定敌人"如何不互相穿模"——boids 排斥力让敌人围着玩家散开成圈，而非叠成一摞 | `enemy_base.gd` → `_compute_separation()`（见 §5） |

设计原则：
- **可预测**：难度曲线挂 `game_manager.game_time`（时间），而非 `upgrade_manager.level`（玩家升级，不可控）。
- **每敌一文件**：每种敌人一个 `.gd`（数值+独特行为都在内）+ 一个 `.tscn`；共用逻辑在基类 `enemy_base.gd`。调某种敌人只动它自己的文件。
- **波次/Boss 节奏是常量表**：调参不动逻辑。

## 2. 数据表（`autoload/enemy_spawner.gd` 顶部）

### 2.1 常量

```gdscript
const MAX_ENEMIES: int = 200            # 同屏敌人上限
const SPAWN_MARGIN: float = 100.0       # 生成圆环到屏幕边缘的额外距离
const SPAWN_EDGE_INSET: float = 50.0    # 生成点离地图墙边的最小距离
```

### 2.2 敌人类型表（共 5 种）

每种敌人的数值**单一数据源**是它自己脚本顶部的 `const CONFIG`（`scripts/enemy/enemy_<kind>.gd`）。下表只是速查，以脚本为准。

字段：`hp` / `speed` / `damage` / `color` / `sprite_size` / `collision_radius` / `damage_area_radius` / `xp_drop` / `show_hp_bar` / `separation_radius`（远程兵 `shooter` 额外有 `shoot_interval` / `shoot_range` / `preferred_range` / `proj_*` / `first_shot_delay`）

| 类型 | 脚本 / 场景 | hp | speed | damage | 颜色 | 尺寸 | xp | 血条 | 独特行为 |
|------|-------------|----|----|--------|------|------|----|------|----------|
| **swarmer** 速攻虫 | `enemy_swarmer.gd` / `.tscn` | 3   | 110 | 3  | 橙 | 14 | 1  | 否 | 无（纯数值） |
| **normal** 普通   | `enemy_normal.gd` / `.tscn`  | 10  | 60  | 6  | 红 | 24 | 1  | 否 | 无（纯数值） |
| **tank** 坦克     | `enemy_tank.gd` / `.tscn`    | 55  | 35  | 12 | 灰 | 40 | 5  | 是 | 无（纯数值） |
| **shooter** 远程兵 | `enemy_shooter.gd` / `.tscn` | 14  | 50  | 6  | 青 | 22 | 2  | 否 | **风筝走位 + 周期发射慢子弹**（覆盖基类 `_get_seek_dir` / `_tick_behavior`） |
| **boss** Boss     | `enemy_boss.gd` / `.tscn`    | 600 | 28  | 25 | 紫 | 72 | 50 | 是 | 无（纯数值；仅由 `BOSS_WAVES` 强制刷出） |

> 只有 `shooter` 需要独特「行为」——它的风筝与开火代码全在 `enemy_shooter.gd`，基类不含远程逻辑。其余 4 种仅靠 `const CONFIG` 的数值差异区分，无需写任何行为代码。

### 2.3 波次表 `WAVES`

按"已过分钟"激活对应阶段（取 `start_min ≤ 当前分钟` 的最后一条）：

| start_min | 可生成原型（加权） | rate_mult | stat_scale |
|-----------|-------------------|-----------|------------|
| 0.0 | swarmer:3, normal:2 | 0.8 | 1.0 |
| 1.0 | swarmer:2, normal:3 | 1.0 | 1.0 |
| 3.0 | swarmer:2, normal:4, tank:1 | 1.2 | 1.1 |
| 5.0 | normal:3, tank:2 | 1.5 | 1.3 |
| 7.0 | swarmer:3, normal:3, tank:2 | 1.8 | 1.6 |
| 9.0 | swarmer:4, normal:3, tank:3 | 2.2 | 2.0 |

- `rate_mult`：乘到基础生成率上。
- `stat_scale`：乘到敌人 **hp 与伤害**上（速度不缩放，保持放风筝手感）。
- **boss 永远不在 `eligible` 里**——只能由 `BOSS_WAVES` 强制生成。

### 2.4 Boss 波表 `BOSS_WAVES`

到点一次性触发（每条只触发一次，靠 `_fired_boss_waves` 记录）：

| trigger_min | boss 数量 | 护卫原型 | 护卫数量 |
|-------------|-----------|----------|----------|
| 3.0 | 1 | swarmer | 12 |
| 5.0 | 1 | normal  | 10 |
| 8.0 | 2 | tank    | 4  |

Boss 聚集在一个环形锚点附近（±40 抖动）一起进场，护卫分散在环形上各点。

## 3. 运行流程

```
enemy_spawner._process(delta)
  ├─ 守卫：game_over / is_paused / 无 player
  ├─ 容器引用惰性重取（is_instance_valid 校验，兼容场景重载/重启）
  ├─ _update_boss_waves()          ← 检查 BOSS_WAVES 是否到点，触发 boss+护卫
  ├─ 同屏上限检查 (MAX_ENEMIES)
  ├─ spawn_accumulator += get_spawn_rate() * delta
  └─ while 累加器 ≥ 1：
       └─ _spawn_a_burst(簇大小)
            └─ 每个位置：kind = _pick_archetype()    ← 当前波次 eligible 加权随机
                └─ _schedule_spawn(kind, KIND_SCRIPTS[kind].CONFIG, stat_scale, pos)
                       │   ↑ CONFIG 仅给「生成预警」读颜色/尺寸（敌人此刻尚未实例化）
                       └─ 预警动画结束 → _on_warning_finished(warn)

_on_warning_finished(warn)
  └─ _spawn_at(warn.spawn_kind, warn.spawn_stat_scale, warn.global_position)

_spawn_at(kind, stat_scale, pos)
  ├─ 敌人 = KIND_SCENES[kind].instantiate()     ← 每种敌人各自的 .tscn
  ├─ enemy.setup(stat_scale)                    ← 必须在 add_child 前调用；setup 内读子类 const CONFIG
  ├─ enemy.global_position = pos
  ├─ enemy.enemy_died.connect(_on_enemy_died)
  └─ enemies_container.add_child(enemy)         ← 此刻 _ready() 触发，读取 setup 写入的配置
```

**生成率公式**：
```
base = minf(1 + game_time/20, 10)      # 0秒=1/s，200秒封顶 10/s
rate = base × 当前波次.rate_mult
```

**环形位置算法**（`get_spawn_position_ring`）：
```
半径 = 屏幕对角线 / 2 + SPAWN_MARGIN     # 对角线保证圆环整圈都在屏外（含四角）
→ 随机角度取点 → 校验是否落在 [inset, MAP_SIZE-inset] 内
→ 最多试 8 次；玩家被逼到角落整圈无合法点时，clamp 兜底
```

## 4. 涉及文件

| 文件 | 角色 |
|------|------|
| `autoload/enemy_spawner.gd` | 生成器主体：`KIND_SCENES`/`KIND_SCRIPTS` 注册表、`WAVES`/`BOSS_WAVES`、环形生成、波次/Boss 调度、加权选型、按类型掉落、`reset()` |
| `scripts/enemy/enemy_base.gd` | **敌人基类**（`class_name EnemyBase`）：共用逻辑——`setup(stat_scale)` 读子类 `CONFIG`、`_physics_process` 追玩家、`_compute_separation` 软分离、击退、`_draw` 血条、`take_damage`→`enemy_died(pos, xp_drop)`；并提供两个可覆盖钩子 `_get_seek_dir` / `_tick_behavior` |
| `scripts/enemy/enemy_<kind>.gd` | **5 个子类**（swarmer/normal/tank/shooter/boss）：各持 `const CONFIG`（单一数据源）；shooter 额外覆盖钩子实现风筝+开火 |
| `scenes/enemy_<kind>.tscn` | **5 个场景**：结构相同（CharacterBody2D + CollisionShape2D + Sprite2D + DamageArea），各自绑定对应子类脚本；运行时由代码按 `CONFIG` 覆盖外观/碰撞 |
| `ui/game_over_controller.gd` | 重启时调用 `enemy_spawner.reset()`，清空已触发 Boss 波记录 |
| `autoload/game_manager.gd` | 只读：`game_time` / `MAP_SIZE` / `player` / `game_over` / `is_paused` / `add_kill()` |

信号契约：
```
enemy_died(position: Vector2, xp_drop: int)
```
唯一发射方 `EnemyBase.take_damage`（所有子类继承），唯一消费方 `enemy_spawner._on_enemy_died`（按 `xp_drop` 循环掉多颗 XP 宝石，带 ±6 抖动防重叠）。

## 5. 敌人间软分离（防重叠）

为防止敌人全部叠到玩家身上，每个敌人在 `_physics_process` 里计算一个**软分离向量**，与"追玩家"的速度混合。这是 boids 群体行为里的 separation，也是吸血鬼幸存者类的标准做法——敌人会围着玩家散开成动态一圈，而非穿模叠成一摞。

> 用的是**软分离（排斥力）**，**不是**开启敌人间物理碰撞。物理碰撞（改 `collision_mask`）在 200 个 `CharacterBody2D` 之间开销大、且会卡顿抖动；软分离更轻、手感更顺。若真要硬碰撞，把 enemy 的 `collision_mask` 从 `64` 改成 `66` 即可（一行），但通常不推荐。

### 5.1 算法（`enemy_base.gd` → `_compute_separation()`）

```
1. 用缓存的 CircleShape2D（半径 = separation_radius）做 intersect_shape 查询：
   collision_mask = 2（敌人层），exclude = [自身 RID]，最多取 SEPARATION_MAX_NEIGHBORS 个邻居
2. 对每个邻居累加排斥力：
   diff     = 自身位置 − 邻居位置            # 指向"远离邻居"
   d        = |diff|
   strength = (separation_radius − d) / separation_radius   # 线性衰减：r 处为 0，越近越强
   steer   += diff.normalized() × strength
3. 完全重叠（d≈0）：按实例 id 取稳定角度给固定推力脱困（不用随机，避免噪声）
4. steer 限幅到 SEPARATION_MAX_FORCE（保留"邻居多/近→推力大"的大小信息，防止超调）
```

在 `_physics_process(delta)` 里与追击速度混合：
```
seek     = 朝玩家方向 × enemy_speed
avoid    = steer × enemy_speed × SEPARATION_WEIGHT
desired  = seek + avoid                     # 限速到 enemy_speed
velocity = velocity.lerp(desired, 1 − exp(−STEER_SMOOTHING × delta))   # 帧率无关低通平滑
move_and_slide()
```

### 5.2 参数（`enemy_base.gd`）

| 参数 | 默认 | 作用 |
|------|------|------|
| `separation_radius` | 各类型 18 / 28 / 26 / 44 / 80（swarmer/normal/shooter/tank/boss，写在各自 `CONFIG`） | 开始排斥的距离，约等于各自尺寸 |
| `SEPARATION_WEIGHT` | 1.0 | 推开强度系数（乘到速度上） |
| `SEPARATION_MAX_FORCE` | 1.5 | 分离合力上限 |
| `SEPARATION_MAX_NEIGHBORS` | 16 | 单次查询的邻居上限（控成本） |
| `STEER_SMOOTHING` | 8.0 | 速度平滑系数（越大越跟手、越小越平滑） |

### 5.3 为什么不抖（设计要点）

集群抖动有三个常见根因，本实现都做了规避：

1. **不用 `1/d` 反距离加权**——它在近距时发散，造成"弹开→seek 拉回→再弹"的振荡。改用**线性衰减 `(r−d)/r`**：边界处平滑归 0、近距有界，邻居进出半径边界也不引起力的跳变。
2. **不每帧 `normalize` 丢大小**——方向会随邻居微小移动而突变。改为**保留合力大小 + 限幅**，力随距离平滑变化。
3. **速度做低通平滑**（`velocity.lerp(desired, …)`）——消除帧间方向突变造成的抽搐；插值系数用 `1 − exp(−k·delta)` 做到帧率无关。

调参：还抖就调大 `STEER_SMOOTHING`；太飘就调小；推开太弱仍堆叠就调大 `SEPARATION_MAX_FORCE` / `SEPARATION_WEIGHT` / 各原型 `separation_radius`；极密集仍抖可调小 `SEPARATION_MAX_NEIGHBORS` 或隔帧计算。

## 6. 关键实现细节（坑位备忘）

1. **各 `enemy_*.tscn` 里 body 与 DamageArea 共用同一个 CircleShape2D 子资源**——原地改 `.radius` 会把两者耦合。`_ready()` 必须 `CircleShape2D.new()` 新建独立 shape 再赋值，不能复用玩家那种原地改写法。
2. **`setup()` 必须在 `add_child()` 前调用**——`_ready()` 在进入树时（add_child 期间）才触发，此时要读取 setup 写入的颜色/尺寸/半径。顺序：`instantiate() → setup(stat_scale) → add_child() → _ready()`。`setup` 现在只收 `stat_scale`，数值来自子类 `const CONFIG`。
3. **`min()` 返回 Variant**，`var x := min(...)` 会被推断为 Variant（本项目把该警告当 error）。生成率计算用 `minf()`（float 专用），从 Dictionary `.get()` 取值流入 `-> float` 时用 `float(...)` 显式转型。
4. **Boss 波一次性触发**：`_fired_boss_waves: Dictionary` 按索引记录已触发的条目；`reset()` 清空它（重启时由 game_over_controller 调用），否则第二局 Boss 波不再触发。
5. **重启容器陈旧**：autoload 的 `_ready()` 只跑一次，`reload_current_scene()` 后缓存的 `enemies_container` 会变成已释放引用。`_process` 里用 `is_instance_valid()` 惰性重取修复。
6. **速度不随 stat_scale 缩放**——只放大血量和伤害，保留放风筝的操作手感。

## 7. 调参指南

全部在 `enemy_spawner.gd`，改完即生效，无需改逻辑：

- **某个类型太强/太弱** → 改对应 `scripts/enemy/enemy_<kind>.gd` 里的 `const CONFIG`（hp/speed/damage/掉落）。
- **整体节奏太快/太慢** → 改 `WAVES[i].rate_mult`（密度）或 `stat_scale`（数值压力）。
- **某原型出现太早/太晚** → 调 `WAVES[i].start_min` 和 `eligible`/`weights`。
- **Boss 节奏** → 改 `BOSS_WAVES` 的 `trigger_min` / `boss_count` / `escort_*`。
- **屏外距离 / 贴墙边距 / 同屏上限** → `SPAWN_MARGIN` / `SPAWN_EDGE_INSET` / `MAX_ENEMIES`。
- **基础生成曲线** → `get_spawn_rate()` 里的 `minf(1 + game_time/20, 10)`。

## 8. 如何扩展

**新增一种敌人**（例如「自爆虫」，临近玩家时引爆）：
1. 新建 `scripts/enemy/enemy_exploder.gd`：`extends EnemyBase` + `const CONFIG`（hp/speed/颜色/尺寸…）。仅需数值差异就此结束。
2. 复制一份 `scenes/enemy_*.tscn` 存为 `scenes/enemy_exploder.tscn`，把根脚本换成 `enemy_exploder.gd`。
3. 在 `autoload/enemy_spawner.gd` 的 `KIND_SCENES` 与 `KIND_SCRIPTS` 各加一行 `&"exploder": preload(...)`。
4. 在希望它出现的 `WAVES[i].eligible` / `weights` 里加入 `&"exploder"`。
5. （若需要独特行为而非只是数值差异）在 `enemy_exploder.gd` 里覆盖基类钩子：例如覆盖 `_tick_behavior` 在 `dist < 引爆半径` 时触发爆炸（参照 `enemy_shooter.gd` 如何覆盖 `_get_seek_dir` / `_tick_behavior`）。**不要再回到基类加 `kind` 分支**——这正是本次拆分要消除的反模式。

**新增一段波次**：往 `WAVES` 数组按 `start_min` 顺序插入一条即可。

**新增一次 Boss 波**：往 `BOSS_WAVES` 加一条，自动按 `trigger_min` 到点触发一次。
